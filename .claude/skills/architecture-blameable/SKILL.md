# Architecture Blameable

This skill owns the ADR (Architecture Decision Record) process for the cover-letter system. It makes every architectural decision traceable (who, when, why) and automatically re-examines decisions against real usage data so the system can pivot when its assumptions stop holding.

**It does not own any runtime behavior.** The cover-letter skill runs letters. This skill records *why* the cover-letter skill is shaped the way it is, and surfaces when that shape should be reconsidered.

## Scope

- Location: `cover-letter/decisions/`
- Naming: `ADR-NNN-<short-kebab-title>.md` (zero-padded 3-digit index starting at `ADR-001`)
- Format: MADR-lite (see template below)
- Triggers: declarative YAML in `cover-letter/decisions/REVIEW_TRIGGERS.yaml`
- Checker: `cover-letter/scripts/check-adrs.ts`
- Make target: `make cl:adr-check` → writes `cover-letter/decisions/adr-review.md`
- Index: `cover-letter/decisions/INDEX.md` (auto-regenerated by the checker)

## When to invoke this skill

1. **User adds a new decision.** Example: "switch from MLflow to DVC". → Author `ADR-NNN-...` superseding the previous MLflow ADR; update `REVIEW_TRIGGERS.yaml`.
2. **A `cl:adr-check` run flagged a stale ADR.** → Read the flag reason; decide *reaffirm*, *update review trigger*, or *supersede*.
3. **Major refactor to cover-letter skill.** → Check whether any ADR predicates the refactor (e.g. changing the 10-phase pipeline requires revisiting `ADR-004`).
4. **Periodic review** (default cadence: every 10 letters or quarterly, whichever comes first). → Run `make cl:adr-check`, audit `adr-review.md`.

## ADR states

| Status | Meaning |
|---|---|
| `Proposed` | Drafted, not yet accepted. Do not rely on it in implementation. |
| `Accepted` | Active decision. System behaves per this ADR. |
| `Superseded by ADR-NNN` | Replaced. Keep the file for history. Add `Supersedes` line to the replacement ADR. |
| `Deprecated` | Decision no longer relevant (e.g. feature removed). Kept for history. |
| `Review pending` | A review trigger has fired; ADR is still Accepted in behavior, but the user has acknowledged the need to re-examine. |

## ADR template (MADR-lite, copy verbatim)

```markdown
# ADR-NNN: <Short title>

- **Status**: Proposed
- **Date proposed**: YYYY-MM-DD
- **Date accepted**: —
- **Owner**: Reebal
- **Supersedes**: — (or ADR-NNN)
- **Review trigger**: see `REVIEW_TRIGGERS.yaml` key `NNN-<slug>`

## Context

<What problem / constraint prompted this decision? 2–4 sentences. Cite concrete
evidence if possible (a specific user complaint, a specific failure mode, a
specific measurement).>

## Options considered

- **A**: <option> — pros: <...>; cons: <...>
- **B**: <option> — pros: <...>; cons: <...>
- **C**: <option> — pros: <...>; cons: <...>

## Decision

<Chosen option + the reason it wins over the others. 2–4 sentences. State
reversal cost explicitly: Low / Medium / High.>

## Consequences

- Positive: <...>
- Negative / accepted tradeoffs: <...>
- Reversal cost: <Low / Medium / High>
- New dependencies introduced: <none / list>

## Validation plan

<How will we know this decision is still right in 3 months? What signal (usage
count, failure rate, user complaint, time spent, cost) should trigger review?>
```

## `REVIEW_TRIGGERS.yaml` format

Each top-level key is the ADR filename-stem (e.g. `002-mlflow-local-for-tracking`). Each entry has a list of `triggers`. Supported trigger types:

| Type | Fields | Auto-evaluable |
|---|---|---|
| `letter_count_threshold` | `value: <int>`, `message: <str>` | Yes — reads `cover-letter/runs.jsonl` |
| `date` | `review_by: YYYY-MM-DD`, `message: <str>` | Yes — compared to today |
| `false_positive_rate` | `source: <critique-file-glob>`, `threshold: <float 0–1>`, `message: <str>` | Yes — greps critique files for flag density |
| `setup_friction` | `threshold_minutes_per_week: <int>`, `message: <str>` | No — surfaced as manual-evaluation prompt |
| `external_event` | `event: <str>`, `message: <str>` | No — surfaced as manual-evaluation prompt |

Example:

```yaml
002-mlflow-local-for-tracking:
  triggers:
    - type: letter_count_threshold
      value: 20
      message: "20 letters logged. Is MLflow UI actually in use? If not, consider simplifying to runs.jsonl."
    - type: date
      review_by: "2026-10-01"
      message: "Half-year review."
    - type: setup_friction
      threshold_minutes_per_week: 30
      message: "If MLflow maintenance exceeds 30 min/week, re-examine."
```

## Checker behavior

`make cl:adr-check` runs `cover-letter/scripts/check-adrs.ts` which:

1. Reads every ADR file in `cover-letter/decisions/` and parses the header block.
2. Reads `REVIEW_TRIGGERS.yaml`.
3. For each auto-evaluable trigger, evaluates against current state.
4. Writes `cover-letter/decisions/adr-review.md` with three sections:
   - **Stale / needs review**: ADRs with ≥ 1 fired trigger. Includes the trigger's message.
   - **Manual-evaluation prompts**: ADRs with at least one non-auto-evaluable trigger.
   - **Current**: ADRs with no fired triggers.
5. Regenerates `cover-letter/decisions/INDEX.md` (a flat table of all ADRs with status, date, title).
6. Exits 0 regardless — staleness is a signal, not a failure. The cover-letter skill's Phase 10 surfaces the review message to the user before finalizing, but never blocks.

## Integration with cover-letter skill

Phase 10 of the cover-letter skill runs `make cl:adr-check` after rendering. If `adr-review.md` contains any **Stale / needs review** entries, the skill surfaces them to the user and pauses with two options:

- **Acknowledge** (letter proceeds; ADR status auto-updates to `Review pending`; a one-line audit entry is appended to the ADR file with today's date).
- **Pivot** (invoke this skill; author a superseding ADR; update `REVIEW_TRIGGERS.yaml`).

## How to author a new ADR

1. Pick the next free number (e.g. `007`).
2. Write to `cover-letter/decisions/ADR-007-<short-slug>.md` using the template above.
3. If the new ADR supersedes an existing one, update the old ADR's Status to `Superseded by ADR-007` and add `Supersedes: ADR-NNN` to the new one.
4. Add a triggers entry in `REVIEW_TRIGGERS.yaml` keyed `007-<short-slug>`.
5. Run `make cl:adr-check` to regenerate the INDEX and verify no parse errors.
6. Commit to the local cover-letter history (the folder is gitignored from the main repo but can be versioned locally if the user has set up a nested git repo).

## What this skill does NOT do

- Does not render cover letters.
- Does not enforce the decisions — enforcement is the cover-letter skill's job.
- Does not modify the 4 critics or the Typst template.
- Does not commit ADRs (the user does, since the folder is gitignored from main).
- Does not invent ADRs — only the user (or their direct proxy this skill) may accept a decision.

## The only rule that matters

**Every architectural decision must be a named artifact with a review trigger.** No silent assumptions. No "we decided this because it felt right". If it shapes how letters are generated, it has an ADR. If the ADR has no review trigger, the decision will ossify and the system will stop pivoting when the world changes. The goal is blame-able, not blame — the artifact exists so we can look back and know why.
